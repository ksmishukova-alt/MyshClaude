# Подключение Supabase к МышМат

Пошагово. Все ключи вводите только у себя (в Vercel и `.env.local`) — никому не пересылайте.

## 1. Создать проект Supabase

1. Зайдите на https://supabase.com → войдите (через GitHub удобнее всего).
2. **New project**. Задайте:
   - Name: `myshmat` (любое)
   - Database Password: придумайте надёжный пароль — **запишите его себе**, он понадобится. Это НЕ ключ API.
   - Region: ближе к вам (напр. Frankfurt / Central EU).
3. Подождите ~2 минуты, пока проект создаётся.

## 2. Выполнить миграции (создать таблицы и данные)

1. В проекте слева: **SQL Editor** → **New query**.
2. Откройте файл `packages/db/setup_all.sql` из проекта, скопируйте **всё** содержимое.
3. Вставьте в редактор и нажмите **Run** (или Ctrl+Enter).
4. Должно выполниться без ошибок. Это создаст все таблицы, функции и демо-данные
   (ребёнок Артём, Маша, банк заданий).

Проверка: слева **Table Editor** → должны появиться таблицы
`child_profiles`, `tasks`, `task_steps`, `daily_sessions` и др.
В `child_profiles` — две строки (Артём, Маша).

## 3. Взять ключи подключения

В проекте: **Settings** (шестерёнка) → **API**. Понадобятся два значения:

- **Project URL** — вида `https://xxxxx.supabase.co`
- **anon public** ключ (длинная строка) — из раздела «Project API keys»

(Ключ `service_role` для MVP не обязателен; если будете использовать — он только для сервера, в браузер не попадает.)

## 4. Прописать ключи

### Локально (чтобы работало на вашем компьютере)

В папке `apps/web` создайте файл `.env.local` (рядом с `.env.example`) и впишите:

```
NEXT_PUBLIC_SUPABASE_URL=https://xxxxx.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=ваш_anon_ключ
```

Файл `.env.local` уже в `.gitignore` — в репозиторий он не попадёт. Это правильно.

Перезапустите `npm run dev`.

### На Vercel (чтобы работало на сайте)

1. Vercel → ваш проект → **Settings** → **Environment Variables**.
2. Добавьте две переменные (те же имена и значения):
   - `NEXT_PUBLIC_SUPABASE_URL`
   - `NEXT_PUBLIC_SUPABASE_ANON_KEY`
3. Сохраните и сделайте **Redeploy** (Deployments → ... → Redeploy).

## 5. Проверить, что всё работает

Откройте в браузере:

```
http://localhost:3000/api/health        (локально)
https://ваш-домен.vercel.app/api/health  (на проде)
```

Должны увидеть JSON:

```json
{ "supabaseConfigured": true, "canQuery": true, "childCount": 2,
  "hint": "Всё работает: Supabase подключён, данные видны." }
```

Если `supabaseConfigured: false` — ключи не подхватились (проверьте имена переменных и перезапуск/redeploy).
Если `canQuery: false` — ключи есть, но миграции не выполнены (вернитесь к шагу 2).

## Как это устроено в коде

- `src/lib/supabase.ts` — клиент; если ключей нет, возвращает `null`.
- `src/lib/data.ts` — фасад: есть Supabase → берёт из БД, нет → моки.
- Поэтому подключение «бесшовное»: те же экраны, просто данные становятся реальными.
- `src/app/api/health/route.ts` — диагностика (этот эндпоинт).

## Обновление: сохранение попыток (миграция 0004)

Добавлена RPC `submit_task_attempt` — сохраняет попытку и пересчитывает
МышРутку на сервере. Чтобы она появилась в вашей БД:

1. Supabase → SQL Editor → New query.
2. Вставьте заново весь `setup_all.sql` (повторный запуск безопасен —
   таблицы через `if not exists`, функции через `create or replace`).
3. Run.

После этого прохождение заданий на сайте начнёт реально писать в БД:
статусы предметов, попытки, и при выполнении всех 4 предметов —
автоматическая выдача МышРутки.

Проверка: пройдите задание «на платформе» до конца, затем посмотрите в
Table Editor таблицы `daily_task_attempts` (появится строка) и
`daily_subject_progress` (статус предмета).
